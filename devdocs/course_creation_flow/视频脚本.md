## 第二章：系统架构全景（1:00 — 2:30）

### 画面描述
- 5 层架构图从下往上逐层构建动画：

```
[存储层] PostgreSQL + pgvector / MinIO / Redis
    ↑
[Celery Workers] knowledge / review / blueprint.synthesis
    ↑
[消息队列] RabbitMQ — 事件驱动链路
    ↑
[API 网关] FastAPI — REST + WebSocket + EventBus
    ↑
[前端] Vue 3 + TS — UploadView / TutorialView
```

### 旁白
> "系统采用五层架构。最底层是存储层——PostgreSQL 负责结构化数据和向量检索，MinIO 存储原始文件，Redis 缓存结果。往上是 Celery Worker 层，按业务域分为五个专用队列。消息通过 RabbitMQ 事件总线串联，形成完整的处理链路。API 层基于 FastAPI 异步框架，前端是 Vue 3 加 TypeScript。"

### 画面切换
- 5 个 Celery 队列名称卡片弹出：`knowledge` / `knowledge.review` / `tutorial` / `blueprint.synthesis` / `low_priority`
- 卡片间用箭头连接显示事件流

### 旁白
> "每个队列有独立的并发和超时策略。比如 ingest 任务 180 秒软超时，extraction 不设限以处理大文档，而 blueprint synthesis 有原子锁保证一个空间同时只跑一个生成任务。"

## 第三章：文档状态机（2:30 — 3:30）

### 画面描述
- 横向状态机动画，7 个状态节点依次亮起：

```
[uploaded] → [parsed] → [extracting] → [extracted] → [embedding] → [reviewed] → [published]
```

- 每次状态切换配一个小勾动画
- extracting 节点短暂闪一下（🔒 原子锁标识）

### 旁白
> "每个文档在系统中经历七个状态。从 uploaded 上传态开始，依次经过 parsed 解析完成、extracting 正在提取、extracted 提取完成、embedding 向量化中、reviewed 审核通过，最终到达 published 已发布。每个状态转换由 Celery 任务完成后自动触发下一个事件。extracting 状态上有一个原子锁，防止同一文档被并发处理。"

### 画面切换
- 状态机下方出现 RabbitMQ 事件链路：
  `file_uploaded → document_parsed → entities_reviewed → embeddings_complete → blueprint_published`

## 第五章：文档解析与分块（5:00 — 6:30）

### 画面描述
- ingest worker 从队列取出任务
- 动画：按格式分流处理——PDF 走 PyMuPDF 通道、DOCX 走 python-docx 通道、MD/TXT 走 charset 检测通道

### 旁白
> "ingest 任务从 knowledge 队列取到消息后，根据文件类型选择解析引擎。PDF 用 PyMuPDF 按物理页提取，同时提取目录大纲；Word 文档用 python-docx 段落级处理；Markdown 和纯文本用 charset 自动检测编码。"

### 画面切换
- 分块动画：一页文本被 LangChain RecursiveCharacterTextSplitter 切成多个 chunk
- 每个 chunk 标注：index_no=1, page_no=3, token_count=245

### 旁白
> "提取后的文本进入分块阶段。使用 LangChain 的递归字符分割器，每页独立分块，保留页面边界信息。每个块带有序号、标题路径、页码和 token 数量。系统最多保留 500 个块，超出部分会被截断并在文档上标记。"

### 画面切换
- 批量 INSERT 动画：chunk 数据写入 document_chunks 表

## 第六章：三步 LLM 知识提取（6:30 — 8:30）

### 画面描述
- 三步流水线动画，垂直排列，从上到下逐步激活：

```
┌─ Step 1: 实体识别 ──────────────────────┐
│ 输入：chunk 文本                          │
│ LLM: ENTITY_RECOGNITION_PROMPT           │
│ 输出：{"entities": [{"entity_name":"..."}]} │
└──────────────────────────────────────────┘
              ↓
┌─ Step 2: 实体分类 ──────────────────────┐
│ LLM: ENTITY_CLASSIFICATION_PROMPT        │
│ 类别：concept / element / flow /          │
│       case / defense                     │
│ 附加：short_definition                   │
└──────────────────────────────────────────┘
              ↓
┌─ Step 3: 关系抽取 ──────────────────────┐
│ LLM: RELATION_EXTRACTION_PROMPT          │
│ 类型：prerequisite_of / related_to        │
│       part_of / example_of               │
│ 条件：仅对 2+ 新实体的 chunk 执行          │
└──────────────────────────────────────────┘
```

### 旁白
> "这是整个流程中 AI 最密集的阶段——三步 LLM 流水线。第一步，实体识别，从文本中提取所有教学相关实体。第二步，将这些实体分类为概念、要素、流程、案例或防御五类，同时生成简短定义。第三步，抽取实体之间的关系——前置依赖、关联、组成、示例。"

### 画面切换
- 去重动画：多个相同 entity_name 合并为一个，带计数器
- 跨文档去重检查：同 canonical_name + 同 domain → 跳过

### 旁白
> "实体在所有 chunk 之间去重，相同名称的只保留一份。同时做跨文档去重——如果另一个文档已经提取过同一实体且属于同一领域，就不再重复录入。去重后的实体批量写入 knowledge_entities 表，初始审核状态为 pending。"

## 第七章：两轮 AI 自动审核（8:30 — 10:00）

### 画面描述
- Round 1 动画：实体卡片队列进入审核引擎，每个卡片被标记为绿色（approved）、红色（rejected）或黄色（uncertain→Round 2）

### 旁白
> "知识提取完成后，进入两轮 AI 自动审核。第一轮，LLM 对每个实体给出 approve、reject 或 uncertain 的判定以及置信度。置信度 0.75 以上的 approve 直接通过，reject 直接拒绝，剩下的进入第二轮。"

### 画面切换
- Round 2 动画：黄色卡片进入更严格的审核
- 最终所有卡片变为绿色（approved）或红色（rejected），无黄色残留

### 旁白
> "第二轮审核更严格，置信度阈值降到 0.60，但依然保证了审核的审慎性。关键设计是——没有任何实体会永久停留在 pending 状态，两轮之后要么 approved，要么 rejected。"

### 画面切换
- PostgreSQL advisory lock 图示：同一空间只允许一个 review 任务执行
- 守护任务时钟动画：每 5 分钟检查一次

### 旁白
> "系统使用 PostgreSQL 的 advisory lock 机制保证空间级别的并发安全，同时有一个 celery beat 定时任务每 5 分钟检查一次是否有卡住的审核需要恢复。"

## 第九章：K-Means 聚类 + LLM 课程生成（11:00 — 14:00）

### 画面描述（分两段）

**第一段：聚类（11:00-12:00）**
- 高维空间中散布 N 个实体点（向量可视化）
- K-Means 算法运行：K = max(4, round(N/7))
- 实体点被分配到不同颜色的簇中
- 均衡化：太小的簇合并，太大的簇拆分
- 过滤步骤显示：CVE 编号、版本号等被标记为灰色"非教学实体"移除

### 旁白
> "蓝图合成是系统最核心的模块。首先，所有 approved 实体的 embedding 向量被加载，系统过滤掉非教学实体——比如 CVE 编号、软件版本号、厂商产品名。然后使用 K-Means 算法对剩余实体聚类，K 值取实体数除以 7 和 4 的较大值，上限 30。聚类后执行均衡化——少于 3 个实体的簇被合并到最近的簇，超过 12 个的簇做二次拆分。"

**第二段：LLM 生成（12:00-14:00）**
- 每个簇依次点亮进入 LLM 生成流程
- 四步生成动画：

```
簇 #1 (12 entities) → [命名] → "Python 安全编码基础"
                              → [阶段规划] → 归入 "基础阶段"
                              → [内容生成] → 600-900 字结构化内容
```

- 内容结构图展示：

```
scene_hook:        "某电商平台因 SQL 注入泄露百万用户数据..."
skim_summary:      [3-5 个要点列表]
full_content:      "完整正文内容... (含 CHECKPOINT 断点)"
code_example:      "PreparedStatement stmt = conn.prepareStatement(...)"
misconception_block: "误区：ORM 框架不能完全防止 SQL 注入"
prereq_adaptive:   { base: "...", if_high: "扩展阅读..." }
```

### 旁白
> "聚类完成后，系统对每个簇执行四步 LLM 生成。第一步，簇命名——给簇起一个教学化的章节标题和教学目标。第二步，阶段规划——将所有章节组织为基础、实践、评估三个阶段，并做动词多样化处理。第三步，生成课程总标题和技能目标。第四步，也是最关键的一步，生成每个章节的结构化内容——包括场景导入 hook、速览要点、600 到 900 字的正文（含 CHECKPOINT 交互断点）、代码示例、常见误区纠正，以及根据前置知识水平的自适应内容。"

### 画面切换
- 数据库写入动画：skill_blueprints → stages → chapters → chapter_quizzes
- 状态 → published

### 旁白
> "所有内容写入数据库后，蓝图状态变为 published，文档也随之标记为 published。同时系统自动派发测验预生成任务。"

### 画面描述

### 旁白

### 画面切换
- 后台处理动画：CHAPTER_REFINEMENT_PROMPT 构建 → LLM 生成 → 内容更新 → refinement_version +1
- Layer 3 联动：测验缓存打叉 → 重新生成动画 → 讨论种子可选更新

### 旁白
> "提交后，系统读取当前章节的元数据和内容摘要，结合教师指令和全局教学约束，调用 LLM 重新生成章节。精调完成后，测验缓存自动失效并重新生成题目，确保题目与更新后的内容保持一致。整个过程对教师透明，几分钟内完成。"

### 画面切换
- 三层方案总览图：

```
Layer 1（生成前）       Layer 2（生成后）       Layer 3（联动）
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ 全局教学约束  │ ──→ │ 自然语言精调  │ ──→ │ 测验重新生成  │
│ 上传时输入    │     │ ✨ 精调本章   │     │ 讨论种子更新  │
│ 注入所有Prompt│     │ 保留版本历史  │     │ 缓存自动失效  │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 旁白
> "这就是 StudyStudio 的教师引导式迭代三层方案。第一层，生成前约束——上传时指定全局教学方向。第二层，生成后精调——用自然语言重写任意章节。第三层，附属内容联动——测验和讨论自动跟随更新。三层配合，让教师既能享受 AI 的自动化效率，又保留了对教学内容的完全控制权。"

## 音频制作建议

| 要素 | 建议 |
|------|------|
| **配音** | AI TTS：Azure 语音合成（中文-普通话-云希/云野）或 ElevenLabs 中文 |
| **BGM** | 科技/未来感电子轻音乐（-18dB 垫底），推荐 Epidemic Sound / Artlist |
| **音效** | 状态切换：轻"叮"声；数据流：数字电子音；完成：柔和提示音 |
| **节奏** | 技术讲解段 150 字/分钟，操作演示段 170 字/分钟，关键概念降速到 130 |

## 画面制作建议

| 要素 | 建议 |
|------|------|
| **架构图** | Figma / Excalidraw / draw.io 导出 SVG → After Effects 动画 |
| **录屏** | OBS 录制 1920×1080，60fps |
| **代码高亮** | Carbon.sh 生成美化截图或 VS Code 录屏 |
| **状态机** | Mermaid 或 Manim 程序化动画 |
| **后期** | DaVinci Resolve / Premiere Pro / 剪映专业版 |
