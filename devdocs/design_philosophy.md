# StudyStudio 核心设计理念

**日期：** 2026-05-02
**状态：** 持续演进

---

## 一、系统定位

StudyStudio 不是"教师工具"或"学生工具"，而是一个**知识课程化与个性化学习平台**。

同一套系统解决多种场景：
- 个人自学：快速甄别知识缺口，生成个人化课程
- 课堂/实训教学：企业讲师建课，学员跟学 + 自适应补漏
- 社会化学习：Fork、改进、分享好的课程
- 入职通道：企业文档 → 缺口诊断 → 快速达到上岗标准

**核心原则：不要让用户选择身份，让用户选择目标。** 身份是行为决定的，不是提前定义的。

## 二、产品设计原则

### 2.1 入口设计：四个目标，不是两个角色

```
┌─────────────────────────────────────────────┐
│  你想做什么？                                 │
│                                              │
│  📄 我有文档，想整理成课程         → 建课入口  │
│  📚 我要系统学习一个主题           → 学课入口  │
│  🔍 我想看看别人做了什么好课        → 社区入口  │
│  ⚡ 我赶时间，只想快速补缺口        → 快速自评  │
└─────────────────────────────────────────────┘
```

同一个用户，今天走第一条路，明天走第二条路，系统不追问"你是谁"。

### 2.2 页面标识：用所有权，不用角色

| 场景 | 页面标识 | 导航文案 | 操作权限 |
|------|---------|---------|---------|
| 自己建的课 | `我的空间 · 编辑` | "我的空间" | 上传/校准/生成/发布 |
| 正在学别人的课 | `学习中 · 来自 xxx` | 进度条 + 原作者署名 | 自评/学习/测验 |
| Fork 后改造的课 | `改编自 xxx · 我的改动` | Fork 来源链 | 加文档/重新生成 |
| 社区发现的课 | `来自社区 · xxx 创建` | 社区溯源 | 加入学习/Fork |

**页面上永远不出"教师/学生"字样。**

### 2.3 交互模式：选择题 + 填空题

> AI 的出题能力 + 用户的经验判断 = 最好的课程

用户永远只做选择题和填空题，不做论述题：
- **选择题：** AI 生成具体、有差异的选项，用户选一个
- **填空题：** 选中的选项有可编辑字段（课时、难度等），用户可微调
- **逃生舱：** 始终提供自由文本输入，但作为次要入口

### 2.4 AI 角色定义

> AI 的角色是"在教学框架、材料证据、用户选择、用户隐性经验约束下，生成可学习、可练习、可评估的课程体验"。

## 三、核心能力

### 3.1 知识课程化（建课）

```
上传材料 → AI 分析+提案 → 用户选择方案 → 经验校准 → Course Map 确认 → AI 生成 → 审阅精调
```

- 5 种方案维度 × 3 套方案（AI 自动选择最合适的维度组合）
- 经验校准：5 道结构化选择题，抽取用户脑中的隐性知识
- Course Map 路由分发：隐性经验精确挂载到对应章节
- 3 种 MVP 教学模板（理论课/任务课/合规课）+ 2 种第二轮模板（案例复盘课/实战课）
- 选择题式精调，AI 定向修改，不重写全部

### 3.2 个性化学习（学课）

```
加入课程 → 自评掌握度 → 抽检验证 → 识别缺口 → 生成个人学习路径 → 学习+测验
```

- 自评清单：系统出题，用户标记"会/不会/不太确定"
- 抽检验证：对标记为"会"的 entity 各出一道验证题
- 缺口课程：只生成用户真正需要学的章节
- 预测性掌握度模型（BKT）为未来方向

### 3.3 社会化学习（社区）

- 多人知识空间 + 邀请码 + 公开分级
- Fork 课程空间：发现别人的课程有缺陷 → Fork → 加自己的文档 → 重新生成 → 学习/分享
- Fork 链溯源："改编自 xxx" 清晰标注原始来源
- 课程讨论、社区发现

### 3.4 快速通道

针对"入职前快速补缺口"这类高压场景：
- 企业文档 → 结构化自评 → 单题抽检每个 entity → 只生成缺口课程
- 不需要集体数据，一个人即可完成全流程
- 时间预估：自评 5 分钟 + 抽检 10 分钟 + 课程生成 + 学习 = 取决于缺口数量

## 四、架构边界

### 4.1 内容层 vs 交付层（不冲突）

```
内容层（建课）：决定教什么、怎么教、用什么案例
  产出：skill_chapters（固定的教学材料）
  变更：低频、高代价、高质量（由用户主动触发）

交付层（学课）：决定学什么顺序、哪里需要重读
  产出：learner_knowledge_states（动态学习状态）
  变更：高频、低代价、个性化（由答题数据自动驱动）
```

- 同样的实体关系图，建课时用户用它定义前置顺序，学课时系统用它生成个人序列
- 交付层**永远不改内容层的章节内容**，只调整交付顺序
- 用户设定的硬前置约束（"第 3 章必须在第 5 章前"）交付层必须遵守

### 4.2 同一个用户的双模式

```
用户 A
  ├─ Space 1（owner）：上传文档 → 校准 → 生成 → 发布
  ├─ Space 2（member）：自评 → 学别人的课
  └─ Space 3（fork from B）：加文档 → 重新生成 → 自己学
```

后端权限在 Space 级别（`require_space_access`）和操作级别（`require_role`），不在全局角色。同一个用户在不同 Space 中可以有完全不同的行为。

## 五、数据策略

### 5.1 自适应学习的冷启动

在没有集体数据的情况下（少数人学一门课）：
- 使用经验校准 + 抽检验证的组合，不依赖 BKT
- 用户自评 + 单题抽检足以完成"会/不会"的二分类

在有集体数据的情况下（热门课被反复学习）：
- BKT 模型对每个 entity 维护 4 参数：P(L₀)、P(T)、P(G)、P(S)
- 集体数据校准 P(G)（猜对概率）和 P(S)（粗心犯错概率）
- 个人数据更新 P(L₀)（初始掌握）和 P(T)（学习概率）
- 新加入者获得更精准的初始判断

### 5.2 数据飞轮

```
用户建课 → 课程被学 → 答题数据 → 优化 gap 推断
    ↑                                    ↓
用户精调 ← 内容问题检测 ← 答题模式分析 ← 测验结果
```

## 六、新手引导策略

### 6.1 第一屏体验

新用户进入系统后不面对空白面板，而是清晰的场景入口 + 一个推荐起点：

- **推荐起点：** 社区精选（让用户先看到"别人做了什么"）
- **快速引导：** 3 步向导（选目标 → 看示例 → 开始使用）
- **帮助中心：** 每个关键页面有 `?` 图标，点击展开场景化帮助

### 6.2 帮助体系

| 用户状态 | 帮什么 | 怎么帮 |
|---------|--------|--------|
| 刚注册 | 了解系统能做什么 | 社区精选 + 场景卡片 |
| 第一次建课 | 理解流程 | 3 步引导 + 内嵌示例 |
| 第一次学课 | 理解自评 | 自评清单 + 示例标记 |
| Fork 别人的课 | 理解 Fork 机制 | 展示"你的改动"vs"原文" |
| 课程生成完成 | 理解下一步 | 审阅引导 + 发布引导 |

### 6.3 措辞规范

| 避免使用 | 改用 |
|---------|------|
| 教师/老师 | 课程创建者（或在具体上下文中用"你"） |
| 学生/学员 | 学习者（或在具体上下文中用"你"） |
| 我的学生 | 学习这门课的人 |
| 布置作业 | 设置测验 |
| 课堂 | 课程/Space |

## 七、竞品定位

StudyStudio 的核心护城河是**知识课程化 + 个性化学习**的双引擎：

| 维度 | 传统 LMS | AI 对话工具 | StudyStudio |
|------|---------|------------|-------------|
| 建课方式 | 手动编排 | 对话生成 | 结构化选择 + 经验抽取 |
| 学习方式 | 固定顺序 | 问答式 | 缺口诊断 + 个性化路径 |
| 教师贡献 | 全部 | 提示词 | 选择题判断 + 隐性经验 |
| 社会化 | 无/Fork | 无 | Fork + 社区 + 溯源链 |
| 企业适配 | 通用 | 通用 | 材料驱动 + 行业经验注入 |
