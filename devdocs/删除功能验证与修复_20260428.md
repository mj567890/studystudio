## 二、发现的 Bug 及修复

### Bug 1：回收站路由被参数路由拦截

**位置：** `apps/api/modules/space/router.py`

**现象：** `GET /spaces/trash` 返回 UUID 解析错误，`DELETE /spaces/trash` 同理。

**根因：** `GET /spaces/trash`（768 行）和 `DELETE /spaces/trash`（813 行）注册在 `GET /spaces/{space_id}`（164 行）之后，FastAPI 将 `trash` 作为 `{space_id}` 参数尝试解析为 UUID。

**修复：** 将两个静态路由移到 `{space_id}` 参数路由之前，删除原位置的重复定义。

### Bug 2：删除影响分析 SQL 未用 text() 包裹

**位置：** `apps/api/modules/space/repository.py:624-639`

**现象：** `GET /spaces/{id}/deletion-impact` 返回 500。

**根因：** `get_deletion_impact()` 中的 SQL 字符串直接传给 `self.db.execute()`，SQLAlchemy 2.x 要求原始 SQL 必须用 `text()` 包裹。仅 `chapter_count` 查询用了 `text()`，其余 7 个均为裸字符串。

**修复：** 所有查询统一改为 `text("SELECT COUNT(*) FROM ...")`。

### Bug 3：清空回收站级联删除失败

**位置：** `apps/api/modules/space/service.py:518-608`

**现象：** `DELETE /spaces/trash` 返回 500，日志 `InFailedSQLTransactionError: current transaction is aborted`。

**根因链（3 层）：**

1. **`chapter_quizzes` 表不存在**：级联删除列表中包含 `DELETE FROM chapter_quizzes`，实际数据库中仅有 `chapter_quiz_attempts`。该查询失败后 `except: pass` 吞掉异常，但 PostgreSQL 事务已中止，后续所有 SQL 报 `InFailedSQLTransactionError`。

2. **`files.file_url` 列名错误**：实际列名为 `storage_url`。导致 MinIO 文件清理前收集 URL 的 SQL 失败。同步修复了 `cleanup_tasks.py` 中的相同问题。

3. **`except: pass` 不 rollback**：虽然后续改用 savepoint 隔离每个 DELETE（`begin_nested()`），但对于不存在的表这仍然不够——每个 DELETE 需要一个独立的 savepoint。

**修复：**
- 删除不存在的 `chapter_quizzes` DELETE
- 补充缺失的级联：`conversation_turns`（先于 `conversations`）、`tutorial_contents`（先于 `tutorial_skeletons`）、`knowledge_relations_temp`、`knowledge_entities_temp`
- `files.file_url` → `files.storage_url`
- 每个 DELETE 用 `async with self.db.begin_nested()` 包裹，失败时 savepoint 自动回滚，不污染外层事务
- 通知表的 DELETE 从 tuple 改为统一 `text()` 对象，简化循环逻辑

### Bug 4：创建空间时不接受 visibility/allow_fork

**位置：** `apps/api/modules/space/router.py:23-26` + `service.py:72-78` + `repository.py:174-181`

**现象：** POST 传入 `{"visibility":"public","allow_fork":true}`，创建后返回 `visibility=private, allow_fork=False`。

**根因链（3 层）：**
1. `CreateSpaceRequest` 缺少 `visibility` 和 `allow_fork` 字段
2. Router handler 只传 `name, space_type, description` 给 service
3. Service `create_space()` 签名无这两个参数
4. Repo `create_space()` INSERT 不含这两个列

**修复：** 四层全部补齐字段，INSERT 用 `COALESCE(:visibility, 'private')` 和 `COALESCE(:allow_fork, FALSE)` 保持向后兼容。

### Bug 5：get_space_with_role 未 SELECT allow_fork/deleted_at

**位置：** `apps/api/modules/space/repository.py:60-82`

**现象：** 数据库存入 `allow_fork=True`，API 始终返回 `False`。

**根因：** SQL SELECT 缺少 `ks.allow_fork` 和 `ks.deleted_at` 列。`_row_to_space()` 用 `getattr(row, "allow_fork", False)` 兜底，默认值 `False`。

**修复：** SELECT 中增加 `ks.allow_fork AS allow_fork` 和 `ks.deleted_at AS deleted_at`。

### Bug 6：DeleteConfirmDialog.vue 文件损坏

**位置：** `apps/web/src/components/DeleteConfirmDialog.vue`

**现象：** 前端构建失败：
1. `vue-tsc`: `computed` 未导入（第二个 `<script setup>` 块中有 import 但第一个没有）
2. `vite`: 文件中存在第二个 `<template>` 和 Markdown 残留文本（AI 对话内容误写入文件）

**修复：** 重写整个组件文件，清理残余内容，修复 import。
