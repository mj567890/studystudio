## 二、A 线：课程生成打磨（P0）

### A1. case 课型（案例复盘模式）

**目标：** 支持事故复盘、管理案例、销售案例等场景

**教学模板：**

```
【案例复盘模式】

1. [背景] — 时间、地点、角色、当时掌握的信息
2. [冲突] — 关键矛盾或两难选择
3. [决策点] — 当事人面临的选择路径（2-3 条）
4. [实际走向] — 实际发生了什么
5. [复盘分析] — 每个决策点的得失、根因分析
6. [可迁移原则] — 从这个案例中提炼的通用原则（3-5 条）
7. [思考题] — 1 个开放式问题：同样的情况你会怎么做？
```

**涉及文件：**
- `apps/api/modules/course_template/` — 新增 case 模板
- `apps/api/tasks/blueprint_tasks.py` — 课型路由
- `apps/api/modules/routers.py` — schema 扩展
- `apps/web/src/components/` — 前端模板预览

**工时：** ~2d

### A2. project 课型（实战演练模式）

**目标：** 支持综合项目、实战演练场景

**教学模板：**

```
【实战演练模式】

1. [任务书] — 项目目标、交付物、约束条件
2. [预备知识检查] — 开始前必须掌握的知识点 + 快速测验
3. [分阶段指引] — 每个阶段的输入、任务、输出、检查标准
4. [常见陷阱] — 每个阶段最常见的 2-3 个坑
5. [里程碑检查] — 每个阶段结束后的自查清单
6. [成果展示] — 如何展示成果、验收标准
7. [复盘模板] — 项目完成后的自我总结框架
```

**涉及文件：** 同 A1

**工时：** ~2d

### A3. 材料版本冲突检测（★ v2.3 新增）

**问题：** 用户上传多份文档时，同一知识点在不同版本中定义矛盾（如 2022 版手册说"1.2MPa 报警"，2024 版说"1.0MPa 报警"）。

**方案：**

```
阶段 1 entity 抽取后：
  ├─ 同一 canonical_name 下的多个 short_definition
  ├─ 做相似度对比（rapidfuzz）
  ├─ 相似度 < 0.6 → 标记为"冲突 entity"
  └─ Course Map 预览页顶部展示冲突列表

冲突展示（仍是选择题）：
  "以下 3 个知识点在不同文档中有不同定义，请选择以哪个版本为准："
  ○ 2022版《设备操作手册》: 1.2MPa
  ○ 2024版《安全规程》: 1.0MPa
  ○ 两版都保留，但在课程中标注差异
```

**涉及文件：**
- `apps/api/tasks/knowledge_tasks.py` — entity 抽取后加冲突检测
- `apps/api/modules/skill_blueprint/` — schema 扩展 + 冲突展示
- `migrations/040_conflict_entities.sql` — 新建表
- `apps/web/src/views/learner/CourseDesignView.vue` — 前端冲突入口

**工时：** ~2d

## 三、B 线：个性化学习（P1）

### B1. 自评清单 + 抽检验证

**场景：** 任何用户加入课程后，第一步不是直接开始学，而是先做"自评 + 抽检"。

**流程：**

```
加入课程 → 知识清单（该课程涵盖的 entity 列表）
         → 逐 entity 标记"会/不会/不太确定"
         → 对"会"的 entity，各出 1 道验证题
         → 做错的拉回"不会"列表
         → 生成只含缺口的个人学习路径
```

**和建课时"经验校准"的关系：** 同一套 prompt 生成逻辑，角色描述不同。复用现有的校准题生成 + entity 绑定能力。

**涉及文件：**
- `apps/api/modules/learner/learner_service.py` — 新增 SelfAssessmentService
- `apps/api/modules/routers.py` — 新增端点
- `apps/web/src/views/learner/` — 自评页面（复用 CalibrationDialog 组件模式）
- `tests/api/` — 新增测试

**工时：** ~2d

### B2. BKT 字段预留

**目标：** 不改现有更新逻辑，只在数据层为贝叶斯追踪做准备工作。

**迁移内容：**

```sql
ALTER TABLE learner_knowledge_states
  ADD COLUMN p_learned       FLOAT DEFAULT NULL,    -- BKT 后验概率（NULL=未启用）
  ADD COLUMN guess_prob      FLOAT DEFAULT 0.15,    -- P(G) 猜对概率
  ADD COLUMN slip_prob       FLOAT DEFAULT 0.10,    -- P(S) 粗心犯错概率
  ADD COLUMN transition_prob FLOAT DEFAULT 0.30,    -- P(T) 学习概率
  ADD COLUMN last_updated    TIMESTAMPTZ DEFAULT NOW();
```

`mastery_score` 继续使用不变。当 `p_learned IS NOT NULL` 时，系统优先使用 `p_learned` 做 gap 判断；否则 fallback 到 `mastery_score`。

**涉及文件：**
- `migrations/041_bkt_fields.sql` — 迁移
- `apps/api/modules/learner/learner_service.py` — 新字段读写
- `tests/unit/` — 验证 fallback 逻辑

**工时：** ~1d

### B3. gap 类型推断（利用已采集数据）

**目标：** 让 GapScan 不再全标 `DEFINITION`。

**方案：**

```
GapScan 现有逻辑：
  mastery < 0.5 → GapType.DEFINITION
  0.5 <= mastery < 0.8 → GapType.MECHANISM

改进后：
  mastery < 0.5 + 查 chapter_quiz_attempts.wrong_entity_ids
    ├─ 该 entity 属于"定义型"知识点（canonical_name 含"概念/定义/原理"）
    │   → GapType.DEFINITION
    ├─ 该 entity 属于"流程型"知识点（含"流程/步骤/执行"）
    │   → GapType.FLOW
    ├─ 该 entity 属于"合规型"知识点（含"规范/标准/红线"）
    │   → GapType.APPLICATION
    └─ 无法推断 → 保持 GapType.DEFINITION（fallback）
```

**涉及文件：**
- `apps/api/modules/learner/learner_service.py` — GapScanService 扩展
- `packages/shared_schemas/enums.py` — GapType 已存在，无需改

**工时：** ~0.5d

### B4. 补洞路径改进

**目标：** 在拓扑排序基础上加入难度权重和去重。

**改动：**
- entity 难度从 `knowledge_entities.metadata` 中读取（已有 `difficulty` 字段）
- 同深度 entity 按难度升序排列（先简单后难）
- 去重：同一条路径中同一 entity 只出现一次

**涉及文件：**
- `apps/api/modules/learner/learner_service.py` — RepairPathService

**工时：** ~0.5d

### B5. 遗忘率按 entity 难度分档

**目标：** 不同难度的 entity 应有不同的遗忘率。

**方案：**

```
core entity（基础概念）    → decay_rate = 0.05（慢遗忘）
concept entity（一般概念）  → decay_rate = 0.10（默认）
advanced entity（进阶内容） → decay_rate = 0.20（快遗忘）
```

在 `bulk_upsert` 时根据 entity 类型自动设置 decay_rate。

**涉及文件：**
- `apps/api/modules/learner/learner_service.py` — MasteryStateService

**工时：** ~0.5d

## 八、风险与对策

| 风险 | 概率 | 对策 |
|------|------|------|
| BKT 字段预留后一直没人用 | 中 | 不影响当前功能，纯增量 |
| 新首页改动过大用户不适应 | 低 | 保留"最近继续"区域，老用户不迷路 |
| case/project 模板质量不稳定 | 中 | 复用 theory/task/compliance 的 6 维质检 |
| 材料版本冲突误报太多 | 中 | 相似度阈值可调，提供"忽略"选项 |
| 三线并行开发互相阻塞 | 低 | A/B/C 无代码交集，可独立推进 |
