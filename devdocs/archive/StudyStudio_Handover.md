# StudyStudio 项目开发交接与状态汇总

## 1. 项目开发目标
StudyStudio 是一个自适应学习平台。后端核心业务基于 Python (FastAPI + SQLAlchemy + PostgreSQL) 开发，前端基于 React/Vue。
项目的核心目标是为学员提供多维度的交互式学习体验，支持复杂的业务逻辑与 AI 辅助能力。

## 2. 当前核心功能进度
目前后端模块 `eight_dim_endpoints.py` 已实现了以下核心功能框架：
* **学习墙 (Wall)**：支持按章节、标签发帖与回复，带有“求助/分享”属性，且集成了 AI 自动引导初答功能。
* **阶段能力证书 (Certificate)**：通过查询学习进度与用户数据，使用 `fpdf` 动态生成 PDF 结业证书并返回文件流。
* **复习提醒 (Spaced Repetition)**：基于艾宾浩斯遗忘曲线机制，自动筛选并提示需要复习的笔记（`due-review`）。
* **AI 整合与主观题批改**：实现了基于 LLM 网关的碎片笔记一键合并（AI Merge），以及根据评分标准（Rubric）对主观题进行自动批阅和 JSON 结构化反馈。

## 3. 极其重要的调试教训与注意事项 (CRITICAL)
在近期的开发与排错中，我们遇到了严重的“连环坑”，接手后续开发时**务必牢记**：

1. **容器代码与本地文件未同步 (Docker Volume Issue)**：
   * **现象**：本地修改了 `.py` 源码并执行了 `docker compose restart api`，但容器内依然报旧代码的错误。
   * **原因**：当前的 `docker-compose.yml` 开发环境可能没有正确配置宿主机到容器的 Volume Mount（数据卷映射），导致本地文件修改后，容器依然在运行旧镜像的代码。
   * **临时解决方案**：在每次修改本地文件后，必须使用 `docker compose cp <本地路径> <容器名>:<容器路径>` 强制覆盖进去，再重启服务。
   * **后续优化方向**：强烈建议修改 `docker-compose.yml`，将 `./apps/api:/app/apps/api` 挂载进去，并配合 Uvicorn 的 `--reload` 参数实现热更新。

2. **Python 字典双大括号陷阱 (`TypeError: unhashable type: 'dict'`)**：
   * **现象**：后端报 500 错误。
   * **原因**：在非字符串格式化的地方，将标准的 JSON 字典写成了双大括号 `{{ "key": "value" }}`。在 Python 中这会创建一个包含不可哈希（unhashable）字典的集合（Set），从而引发崩溃。
   * **对策**：在写普通字典或列表推导式时，严格使用单大括号 `{}`。仅在 f-string 或 `.format()` 中需要转义大括号时才使用 `{{}}`。

3. **慎用 `sed` 或简单的正则进行全局符号替换**：
   * **现象**：为了修复大括号，使用了替换脚本全局修改，导致正常用于闭合的 `}` 也被错误计算，引发全局 `SyntaxError`（代码少括号）。
   * **对策**：如果需要批量修改代码，应使用基于 AST 解析的代码格式化工具，或者在 Python 脚本中采用极其精确的正则表达式进行闭环验证（如验证左右括号数量平衡）。

## 4. 协作与代码生成习惯
* **精准备份与修改**：当前习惯于使用 Bash 中的 Python Heredoc (`python3 - << 'PYEOF'`) 来精准定位并修改文件内容（基于 pathlib 和 re），以防手动复制粘贴大段代码造成缩进或字符丢失。
* **用数据说话**：一旦遇到报错，不再盲猜。必须先执行 `docker compose logs api --tail=50` 抓取真实报错堆栈，并直接打印容器内（或本地）的真实代码以核对当前运行的逻辑。

## 5. 后续开发方向
1. **基础设施修复**：优先修复 Docker 的本地代码挂载热更新能力，大幅降低 Debug 成本。
2. **前后端联调**：确保前端请求接口时的 UUID 格式、Token 验证逻辑与后端的期望严格一致。
3. **AI 模块强化**：针对 AI 主观题批阅和 AI 回复，进一步增加异常捕获和重试机制，防止因大模型返回格式不稳定导致业务中断。
