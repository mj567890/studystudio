# 删除功能设计讨论

> 日期：2026-04-28
> 讨论主题：资料库删除、回收站、Fork 文档共享、公开分级
> 参与：用户 + AI 编程助手

---

## 一、讨论起点——删除资料库后的运行逻辑

### 发现
- 当前代码库**没有 `DELETE /spaces/{space_id}` 端点**
- `knowledge_entities` 和 `documents` 表的 `space_id` 字段**无 FK 约束**——删除 space 后这些数据成为僵尸数据
- 事件系统没有 `space_deleted` 事件，Celery 任务不响应空间删除
- `skill_blueprints.space_id` 为 NOT NULL（需手动处理），`tutorial_skeletons.space_id` 为 ON DELETE SET NULL
- MinIO 文件不会被清理

### 存量僵尸数据诊断 SQL
详见方案文档。

## 二、回收站方案讨论

### 用户主张
不要直接硬删除。做回收站：删除→进回收站→回收站内彻底删除。回收站有清空和还原功能。

### 结论

**方案：标准软删除模式**

```sql
ALTER TABLE knowledge_spaces
  ADD COLUMN deleted_at TIMESTAMPTZ,
  ADD COLUMN deleted_by UUID REFERENCES users(user_id);
```

状态机：
```
正常 → DELETE → 回收站 → POST restore → 正常
                        → DELETE permanent → 彻底删除
                        → DELETE trash → 批量彻底删除
```

**对课程的影响：课程自然不可见，不需要单独处理。** 因为课程列表查询 JOIN `knowledge_spaces`，加了 `deleted_at IS NULL` 过滤后自然会隐藏。还原后自动恢复。`skill_blueprints.space_id` FK 关系保持（行还在）。

## 三、Fork 文档共享问题

### 用户发现的关键问题
课程有"章节源文档溯源"功能——显示知识点来源文件的哪些页。如果 Fork 不带文档，这个功能在 Fork 的课程中就失效了。

### 调研结论
- Fork 当前**不复制文档**（不复制 `documents`、`document_chunks`、MinIO 文件）
- 溯源功能是通过在 `document_chunks` 中实时语义搜索实现的，不是查预存的引用关系
- 没有文档 chunks = 溯源完全失效

### 用户方案（被采纳）
**共享文档池 + 引用计数**，避免存储爆炸。

```
源空间 A（owner=A，被 B、C fork）
├─ documents（唯一存储）
├─ entities → B、C 复制了副本
└─ blueprint → B、C 复制了副本

Fork 空间 B/C 不复制文档，通过 space_document_access 表引用 A 的文档
溯源搜索时跨空间检索被共享的文档
```

**删除规则：**
| 操作 | fork_count > 0 | fork_count = 0 |
|------|:---:|:---:|
| 源用户删除空间 | 只对删除者不可见（软删除），文档保留 | 允许硬删除，全量清理 |
| Fork 用户删除空间 | 级联清理自己副本，不动源文档 | — |

### 补充讨论的问题

**链式 Fork（A ← B ← C）：**
C 的溯源需要能追溯 A 的文档。Fork 时传递文档引用链：
```sql
-- C fork B 时：
INSERT INTO space_document_access (space_id, document_id, source_space_id)
SELECT 'C', sda.document_id, sda.source_space_id
FROM space_document_access sda WHERE sda.space_id = 'B';
```

**单个文档删除：**
文档也走软删除（`documents.deleted_at`）。溯源查询过滤 `deleted_at IS NULL`。引用计数判断是否可彻底物理删除。

## 四、学生角色与参与机制

### 调研结论

系统采用**自组织学习空间模型**，不是传统师生模型：

- `owner/admin/member` 三级空间角色
- **学生 = member**，加入空间（邀请码/公开加入）即可学习
- **学生不需要 Fork**——Fork 是给想复制课程二次开发的人用的
- 学生学习数据（进度、掌握度、笔记）存在个人名下
- 社交内容（讨论帖、学习墙）存在课程空间名下

### 对删除的影响

**硬删除空间时：**
- `learner_knowledge_states` 随 entity CASCADE 删除——学生 30 小时的学习记录清零
- `note_entity_links` 随 entity CASCADE 删除——笔记失去知识关联
- `course_posts`、`wall_posts` 随 space CASCADE 删除

**建议：** 硬删除确认弹窗需告知学生数据影响范围。建议 7 天通知期让学生导出学习记录。

## 五、章节/实体删除 vs 空间删除

### 用户纠正

两个层级的删除应该分开处理：

| 层级 | 硬删除条件 | 前提 |
|------|-----------|------|
| 章节/实体 | 用户确认即可 | 老师编辑课程，该删就删 |
| 整个空间 | fork_count = 0 | 需检查是否被 fork |
| 整个空间（被 fork） | 不可硬删除 | 只能对删除者不可见 |

**章节硬删除时：**
- 当前有 6 个衍生表用 `chapter_id VARCHAR` 引用章节，全无 FK 保护
- 需要在删除时手动级联清理：`chapter_quizzes`、`chapter_progress`、`chapter_reflections`、`chapter_quiz_attempts`、`course_posts`（v016 无 FK）、`wall_posts`
- 告知学生："课程章节已被老师删除"

## 六、公开课程的分级设置

### 当前状态
- visibility 只有 `private`/`shared`/`public`
- `shared` 后端支持但前端无入口（未启用）
- Fork 权限**绑定在 public 上**——公开了就能 fork，不能独立控制

### 讨论结论

**应该拆分为两级独立设置：**

```
visibility（可见性）：         allow_fork（允许 Fork）：
  private  仅邀请               true   允许
  shared   链接分享              false  禁止（默认）
  public   发现页可见

组合：
  public + allow_fork=false  → 公开课，可学不可 Fork
  public + allow_fork=true   → 公开课，可学可 Fork
  private + allow_fork=true  → 无意义，不可见就谈不上 fork
```

## 七、告知义务

### 设置公开 + 允许 Fork 时的确认弹窗

必须明确告知：
1. 上传的文档将共享给 Fork 用户
2. 被 Fork 后无法强制删除已共享的文档
3. 因主动公开导致的内容扩散，系统不承担责任
4. 用户须勾选"我已阅读并理解"才能确认

### 硬删除空间时的影响告知

展示当前学员数、学习记录数、讨论帖数、测验成绩数，以及删除后的不可逆影响。

## 八、Fork 同步功能

### 当前状态
- Fork 是一次性快照复制
- `fork_from_space_id` 字段已预留但同步逻辑未实现
- 订阅通知机制就绪（`space_subscriptions` + `blueprint_merged` 事件），但只通知不实际同步

### 讨论结论

**增量同步 + 共享文档池天然互补：**
- 源作者新增文档 → fork 用户只需加 `space_document_access` 记录，不用复制文件
- 同步时只拉取新增内容，已有内容以 fork 用户为准
- 已有章节/实体的 `chapter_id`/`entity_id` 不变 → 衍生数据完全不受影响

---

## 九、讨论区范围

### 当前状态
- 讨论区按 `space_id` 隔离
- Fork 后新空间有独立的空讨论区
- Fork 用户和源空间学生不共享讨论区

### 讨论结论

**保持 per-space 隔离。**

但新增可选功能：Fork 空间章节页可"引用显示"源空间对应章节的讨论（只读），解决 Fork 用户面对"空城"的问题。

---

## 十、关键设计原则总结

1. **回收站标准软删除**：空间级别，30 天过期
2. **共享文档池**：不复制文档，引用计数控制删除
3. **Fork 不影响源空间删除**：有 fork 时源不可硬删除
4. **公开分两级**：visibility 和 allow_fork 独立
5. **章节可变，空间被保护**：章节/实体该删就删，空间受 fork 约束
6. **尽告知义务**：设置公开、允许 fork、硬删除时均弹窗确认
7. **学生数据保护**：硬删除前通知期，确认弹窗告知影响
8. **讨论区隔离**：per-space，可选源课程讨论引用
9. **增量同步**：纯增量，不覆盖已有，不产生垃圾数据
10. **永不删除已发布章节原则（修正）**：章节可以删，老师说了算，系统尽告知义务
