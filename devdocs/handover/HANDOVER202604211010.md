## 二、本次会话完成工作（2026-04-20）

### Blueprint V2 质量三大修复

**P0 — 章节内容覆盖修复（T1 根因）**
- 问题：`chapter_contents` 用 title 做 key，同名章节后写覆盖先写，导致内容污染
- 修复：`blueprint_tasks.py` 中将 key 从 title 改为 cluster_id，彻底消除碰撞风险
- 影响：websecret v11 的 ch6/ch7 内容污染将在下次重跑后消失

**P1 + P2 合并 — Stage 规划 + 动词多样性（新 LLM 轮）**
- 新增 `STAGE_PLANNING_PROMPT`：聚类后调一次 LLM，一次完成：
  1. 按学习难度分组阶段（foundation → practice → assessment）
  2. 检测同词头章节（>15%）并重命名，动词多样性池：识别/理解/应用/分析/评估/构建/设计/防御/检测/配置/实施/调试/优化/比较/解释/演示

**P3 — 簇大小均衡（`_rebalance_clusters`）**
- 超大簇（>12）子聚类拆分（KMeans，目标 7）
- 过小簇（<3）合并到最近邻；支持多轮迭代
- 确保章节规模均匀，避免 2~18 实体极差

**章节内容质量提升**
- `full_content` 要求从 400-600 字扩展到 600-900 字
- 新增 `code_example` 字段：涉及编程/命令行时必须提供完整可运行示例（`<pre><code class="language-xxx">` 格式）
- 高阶提示（if_high）扩展到 100 字

### LLMGateway 根因修复（`llm_gateway.py`）

- 旧方法（v4）：task 入口重置单例（绕过而非根修）
- 本次真修：`_lock` 改为惰性创建，按 running loop 绑定（`_get_lock()`）
- 加载路由时创建独立引擎（不复用全局 `async_session_factory`），彻底解决 celery prefork 下的 event loop 冲突
- 新增分级超时：`tutorial_content` / `blueprint_synthesis` / `coherence_eval` 用 120s，其他 30s

### routers.py — space_id 全线贯通

- 教程获取（/tutorial）、对话创建/列表（/conversations）、聊天（/chat）、修复路径（/repair-path）均支持 `space_id` 参数
- 对话列表增加 `last_message` 预览字段（截取前 80 字符），limit 从 50 升 100
- `list_teaching_spaces` 改用 `space_members` 关联查询（不再限于 owner，加入的 space 也能看到）

### 章节完成自动同步实体进度

- `mark_chapter_progress` 标记完成时，自动：
  1. 快照到 `chapter_progress_entities`（幂等，ON CONFLICT DO NOTHING）

- 替代旧的手动诊断路径，完成即更新掌握度

### admin/router.py 改进

- 新增 `GET /admin/courses`：列出所有有 blueprint 的课程，含章节数、内容完成数统计（LATERAL JOIN）

### Phase 3a — 课程讨论区

#### 产品逻辑
- 每门课程（space）有一个讨论区
- 帖子可绑定到某个章节，也可是全局讨论
- 帖子类型：笔记 / 求助 / 讨论
- 支持回复（一层，不做无限嵌套）
- 入口：学习页章节旁「讨论」按钮；侧边栏「讨论」菜单聚合所有课程

#### 需要新建的数据表

```sql
CREATE TABLE course_posts (
    post_id       UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    space_id      UUID NOT NULL REFERENCES knowledge_spaces(space_id) ON DELETE CASCADE,
    chapter_id    UUID REFERENCES skill_chapters(chapter_id) ON DELETE SET NULL,
    user_id       UUID NOT NULL REFERENCES users(user_id),
    post_type     VARCHAR(20) NOT NULL DEFAULT 'discussion'
                  CHECK (post_type IN ('note', 'question', 'discussion')),
    title         VARCHAR(255),
    content       TEXT NOT NULL,
    reply_count   INTEGER NOT NULL DEFAULT 0,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE course_post_replies (
    reply_id   UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    post_id    UUID NOT NULL REFERENCES course_posts(post_id) ON DELETE CASCADE,
    user_id    UUID NOT NULL REFERENCES users(user_id),
    content    TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 本次会话更新（2026-04-20）— RabbitMQ 修复 + 页码显示修复

### RabbitMQ UUID 队列积压根治
- 根因：`backend="rpc://"` 导致每个 task 在 RabbitMQ 创建 UUID 临时队列，发完不读结果导致积压
- 修复：`tutorial_tasks.py` backend 改为 `redis://redis:6379/1`，结果写 Redis 按 `result_expires=3600` 自动过期
- 存量两个积压队列（164条、3条）已通过 `rabbitmqadmin delete queue` 手动清除
- 说明：`celeryev.*` 和 `.pidbox` 队列有消费者、消息为 0，是正常心跳队列，无需处理
