## v2.1 → v2.2 变更（2026-05-01，第二轮专家反馈）

### 三位专家的反馈方向

**专家 A（整体肯定 + 小幅微调）：** 5 条小建议（模板适应性兜底、认知负荷递进、学生数据注入时机、向后兼容 UX、锦上添花但非必须）

**专家 B（LLM 工程风险警告）：** 3 个工程级高风险点（Course Map 黑盒化、上下文过载 Lost in the Middle、"干瘪材料"与 9 维检查死锁）

**专家 C（尖锐批评）：** v2.1 在产品定位上比 v2 更危险，"精致的工程掩盖了空心的内核"，动态访谈被有意识地删除是最严重的问题

### 决策逻辑

对每条反馈做了独立的"必须改 / 建议改 / 暂不改"判断：

**必须改（3 项）：**
1. 动态访谈/经验校准 — 这是产品差异化核心，不加则与 NotebookLM 等无本质差异
2. Course Map 教师预览 — 两位专家独立提出，防止 Token 浪费
3. 第一轮范围缩减 — 5 种课型 + 9 维检查在 2-3 周内吃不下

**建议改（4 项）：**
4. SOURCE 绑定 chunk_id — 10 分钟改动，后期事实追溯必需
5. 场景密度改为结构化字段检查 — 避免关键词误判
6. Bloom 检查直接用结构化字段 — 零成本，避免中英文匹配问题
7. 认知负荷递进检查 — 提升 Course Map 校验质量

**暂不改（2 项）：**
8. 分层生成（Pipeline Generation）— YAGNI，先验证核心闭环
9. Safe Hallucination — 正确解法是让教师提供真实案例（通过经验校准），而非让 AI 编造

### 变更汇总

| 组件 | v2.1 | v2.2 |
|------|------|------|
| 阶段数 | 5 | 5（但阶段 2 拆为 2a/2b/2c） |
| 方案类型 | 5 种维度 AI 自选 | 不变 |
| 经验校准 | 无 | **新增阶段 2b：5 道选择题** |
| Course Map 预览 | 无 | **新增阶段 2c：大纲树预览+确认** |
| 课型 | 5 种 | **第一轮 3 种（theory/task/compliance）** |
| 质量检查 | 9 维 | **第一轮 6 维** |
| SOURCE 标记 | entity_id | **entity_id:chunk_id** |
| Bloom 检查 | 中文文本匹配 | **结构化 verb/bloom_level** |
| 认知负荷递进 | 无 | **Course Map 校验增加** |
| 场景密度检查 | 关键词匹配 | **结构化字段为主，关键词 fallback** |

### 为什么 v2.1 遗漏经验校准是危险的

v2 缺动态访谈，是**显然的缺口**——一眼能看出来"教师贡献的信息太少"。

v2.1 用一堆精巧的工程设计（5 种维度、Course Map、9 维质检）把这个缺口**遮起来了**——看起来很全面，实际上核心信息源没变。

教师在整个 v2.1 流程中贡献的真实信息量 < 10 bit（方案选择 1.6 bit + 4 个可被 AI 推测的参数），全部是显性维度。材料里没有的真实事故、隐性经验、老员工才知道的坑——AI 永远不会知道。

而这些恰恰是企业付费做内训的真正理由。否则让员工直接读 SOP 就行了。

经验校准是唯一的非对称优势——它需要一套独特的提问方法论，随着教师答得越多提问越准，形成数据飞轮。对手抄不走，因为他们要么没意识到这是核心，要么产品架构（"上传即出课"）不支持。

## v2.2 → v2.2-p3 变更（2026-05-01，第三轮工程补丁）

### 四位专家的反馈方向

- 专家 A：整体肯定 + 小建议（校准题 Prompt 优化、Bloom 标签）
- 专家 B：指出了 Context Bleed 工程风险 —— **最关键的一条**
- 专家 C：7 条细节建议（confidence_score、重新规划行为、案例分配、精调限制标注、材料版本冲突占位、移动端交互、成本估算）
- 专家 D：3 条补丁（校准题质检、重新规划原因选择、经验校准进入验收）

### 决策：必须改 4 项，建议改 4 项，暂不改 3 项

**必须改（不改会出工程事故或产品退化）：**

| # | 反馈 | 风险 | 修改 |
|---|------|------|------|
| 1 | Context Bleed：全局校准数据广播到每章 | 第 1 章理论课强行插入操作红线 | 路由分发：Course Map 增加 calibration_routing，内容生成只收章节级数据 |
| 2 | 经验校准流于形式 | 教师全选"不清楚"，校准全空但系统以为"已校准" | confidence_score + 低分预警（<0.4 提示邀请补答） |
| 3 | 重新规划行为不明确 | 点了等于没点，AI 给出几乎一样的大纲 | 反例注入 + 原因选择（4 个选项）+ 3 次上限 |
| 4 | 经验校准题质量无检查 | AI 生成太泛的题，不需要经验就能选 | 5 条自动质检规则 |

**建议改：**

| # | 反馈 | 修改 |
|---|------|------|
| 5 | 重新规划无原因 | 4 个原因选项（仍是选择题） |
| 6 | Course Map 缺 Bloom 标签 | 每章显示 Bloom 层级 |
| 7 | 缺成本估算 | Token 估算表 + 关键指标埋点 |
| 8 | 第二轮精调智能度无说明 | MVP 范围表增加已知限制 |

**暂不改/记录：**

| # | 反馈 | 理由 |
|---|------|------|
| 9 | 材料版本冲突检测 | v2.3 待补，当前用 confidence_score 缓解 |
| 10 | 移动端拖拽交互 | Web 优先，移动端改为"勾选+编号" |
| 11 | 跳过率阈值硬编码 | 改为"观察 4 周，按 P75 确定" |

### 最关键的设计变更：路由分发

原始 v2.2 设计中把教师所有经验校准数据全局注入到每章 prompt。LLM 的"讨好型人格"会导致它在不合适的章节强行引用红线/案例，造成内容逻辑割裂。

修正后的数据流：

```
教师 5 道题 → experience_calibration (全局)
    ↓
COURSE_MAP_PROMPT 收到全局校准
    ├ 生成章节结构
    └ 做路由分配 → calibration_routing（每章子集）
         ↓
CHAPTER_CONTENT_PROMPT 只收 {chapter_calibration}
    ├ 理论课 → 仅误区
    ├ 任务课 → 真案例 + 痛点
    └ 合规课 → 红线 + 真案例
```

这个改动不影响前端，不影响 API schema，仅影响后端 prompt 拼接逻辑。

## v2.2-p3 → v2.2-p3 第四轮补丁（2026-05-01，实现边界条件论证）

### 反馈来源
三位专家在 v2.2-p3 完成后指出了实现层面的边界条件——不是架构问题，是"开发前再花 2 小时想想"的细节。

### 采纳 6 项，全部写入方案

| # | 反馈 | 风险 | 修改 |
|---|------|------|------|
| 1 | **Routing Loss（路由遗漏）** | LLM 生成 Course Map 时静默丢弃校准数据，红线"蒸发" | 新增 Zero-Loss Check（校验规则第 7 项），不通过时单独发轻量 LLM 重新分配，不动 Course Map 结构 |
| 2 | **Case facet 分配不明确** | 同一案例在 2 章重复相同文字，学生感觉啰嗦 | calibration_routing 的 cases 增加 `facet` 字段（scenario/consequence/analysis/transfer），不同章用不同切面 |
| 3 | **低置信度应改变 AI 行为** | AI 看到 calibration 为空会自己脑补案例——企业培训最危险的失败模式 | CHAPTER_CONTENT_PROMPT 增加 confidence-driven behavior：<0.4 时保守生成，不编造、不冒充教师确认 |
| 4 | **重新规划原因→不同 prompt 策略** | 4 个原因都触发同一"完整重生成"，教师反复试错 | 5 个原因各映射不同策略（重排序/调粒度/调权重/标记模式/完整重生成） |
| 5 | **校准题质检阈值太宽松** | max > 0.9 意味着只要 1 个选项不是原文就过关 | 改为"≥50% 选项与原文相似度 < 0.6"+"选项多样性检查（至少 2 类倾向）" |
| 6 | Bloom 标签文字→渐变符号 | 教师扫一眼判断递进太慢 | ●○○○○○ 六段渐变，1 分钟判断前松后紧 |

### 元层面提醒
**文档优化边际收益已很低。该开工了。** 四轮迭代从 v2 → v2.1 → v2.2 → v2.2-p3，每次都是线性收敛而非推翻重来。让真实材料和真实教师碰一碰，第一批数据回来再决定方向。

## 工程落地提醒（2026-05-01，开工前最终确认）

### 架构状态

四轮专家修订（30 条建议，采纳 24 条，暂缓 4 条，记录 2 条），架构已收敛。以下不是架构问题，是开发中需要管理预期的工程现实：

### 5 条落地前须知

| # | 提醒 | 应对 |
|---|------|------|
| 1 | **API 并发与超时**：15 章课程 ≈ 200K tokens，串行 >10 分钟，并行触发 rate limit | Celery 重试机制 + 前端进度条动画（"正在查阅安全操作规范…"模拟思考文案） |
| 2 | **Prompt 脆弱性**：整合生成 prompt 很长，LLM 容易丢三落四（JSON 正确但忘了 SOURCE 标记等） | 预算 2-3 周专门做 prompt 调优 + 回归测试集（Eval），不要假设第一版 prompt 就是终版 |
| 3 | **题型库行业适配**：5 道校准题（真痛点/真案例/真误区/优先级/红线）对某些行业不完全对应（如销售类的"客户类型识别+异议处理"、IT 运维的"故障排查路径+临时变通方案"） | 第一批客户跑完后识别适配差的行业，再加题型库分支，不用提前做 |
| 4 | **路由分发首次成功率**：LLM 同时做"章节切分+校准路由"两件事，稳定性有限。Zero-Loss Check 是兜底 | 监控首次失败率，如果 >30%，考虑把路由拆成单独 prompt 调用（多 1 次 LLM 成本，换稳定性） |
| 5 | **核心验证指标**：不要追求所有指标都漂亮 | **聚焦一个指标：教师经验是否被正确路由、正确使用，并且明显提升课程真实感与可学性**。这个跑通，平台就有差异化 |

### 文档状态

**不再改架构。开工。**
