### 🔴 高风险问题（2 项）——触发退回

**R-01：M05 知识抽取工作量严重低估**

- **当前估算**：4 天
- **问题分析**：M05 实际包含以下工作量：
  - 四步分层管线实现（实体识别/分类/关系/流程重建）
  - JSON Schema 强校验 + 业务规则软校验（双轨校验）
  - 3 套 Prompt 模板工程与调试
  - FewShotManager 领域示例管理
  - 三级降级策略（warn 70% / pause 60% / stop 50%）
  - 审核队列推送逻辑
  - 单元测试覆盖（至少 4 个 step × 3 场景 = 12 用例）
- **实际工作量**：7-8 天
- **低估影响**：M05 是 M06（归一化）的硬前置，低估会导致 M06 延迟，进而阻塞 M11/M14，最终影响 MS-2 以后所有里程碑
- **修订要求**：将 M05 拆分为 M05a（抽取管线 + 校验，5 天）和 M05b（FewShotManager + 降级，2 天），总计 7 天。M06 等待 M05a 完成即可启动（不需等待 M05b）

---

**R-02：M15 教学对话未拆分，单模块风险过高**

- **当前估算**：4 天（不含拆分）
- **问题分析**：M15 当前包含以下相互独立的高复杂度组件：
  - `classify_query_complexity`（复杂度分类器，移除 no_prior_context 信号后需要重新验证）
  - `normalize_rrf_score()`（RRF 归一化，量级问题修复）
  - `TEACHING_STRUCTURED_PROMPT_SUFFIX` + JSON 解析（结构化输出规格）
  - `TeachingChatService.chat_and_prepare`（主体逻辑，乐观锁 + BackgroundTasks）
  - `DiagnosisWriteService`（乐观锁合并，两种合并策略）
  - `_run_diagnosis_update`（独立 session 后台任务，R1 修复）
- **风险**：任何一个组件出问题会阻塞整个 M15，导致 MS-4 里程碑延迟。各组件可独立开发和测试，拆分可以提前暴露问题。
- **修订要求**：拆分为 M15a（LLM Gateway + 结构化输出 + RRF 归一化，2 天）和 M15b（TeachingChatService 主体 + 诊断写入，3 天）。M15b 依赖 M15a。

---

### 🟡 中风险问题（1 项）——建议修订

**R-03：M18 事件管理被排到阶段 5，但事件总线在 M04-M15 全程被依赖**

- **问题分析**：`event_bus.publish()` 在 M04（document_parsed）、M07（learner_state_updated）、M11（skeleton_generated）等模块均有调用。若 EventBus 基础设施（aio-pika 连接封装、幂等表初始化）未在这些模块之前就绪，会导致早期开发和测试时事件发布失败，影响调试效率。
- **修订建议**：将 EventBus 核心基础设施（`core/events.py`）并入 M00，与数据库、MinIO 在同一阶段初始化。M18 保留用于各模块的事件消费者注册和 DLQ 配置。
- **影响**：M00 工作量增加 1 天（3 天 → 4 天），换来 M04-M15 全程事件调试无障碍。

## 二、资源分配合理性审查

### ✅ 通过项

- **BE-Know 高负荷可接受**：M04→M05a→M05b→M06 虽为串行，但知识模块本身存在技术依赖，串行是必要的，不是资源浪费。M05b 可与 M06 并行（M05b 不是 M06 的硬前置）。
- **关键路径工程师未过度分配**：BE-Chat（M14+M15a+M15b）共 8 天，与总工期比例合理。

### 🟡 需要关注项

**FE 工期偏紧（10 天完成学习端）**
- 学习端涉及 8 个核心页面（学习画像、题库、漏洞报告、补洞路径、教程、对话、注册/登录）加上状态管理和接口对接
- 10 天在有完整接口文档和设计稿的前提下是可行的，但缺少任何一项都会导致超期
- **建议**：延长至 12 天，并要求 INT 在 Day 30 前完成接口文档（`contracts/*.yaml`），确保 FE 不因等待接口文档而浪费时间

**QA 时间窗口未明确**
- 当前计划未为 QA 分配具体时间窗口，存在测试被挤压到最后 2-3 天的风险
- **建议**：QA 采用"模块完成即测试"模式，每个模块通过 CR 后 QA 立即介入，与下一模块开发并行，避免集中测试风险

## 三、风险控制完备性审查

### ✅ 覆盖充分的风险

| 风险类型 | 控制措施 | 评价 |
|---|---|---|
| LLM API 不可用 | 三级降级（主模型→备用→模板回答） | ✅ 充分 |
| 并发幂等（骨架重复创建） | Redis 分布式锁（B2）+ DB UNIQUE 约束（D2） | ✅ 双重保护 |
| 向量查询错误 | pgvector 编解码器注册（C2）+ 集成测试验证 | ✅ 有验证手段 |
| 诊断写入丢失 | BackgroundTasks + 独立 session（D3+R1） | ✅ 根因已修复 |
| 文档截断信息丢失 | 截断前记录 original_count（R2） | ✅ 已修复 |
| 知识图谱循环依赖 | topological_sort_safe + 单元测试（B1） | ✅ 已验证 |

### 🟡 缺失的风险控制

**数据库迁移失败无回滚方案**
- `001_initial_schema.sql` 是单文件全量脚本，没有对应的 `down` 脚本
- 建议引入 Alembic 管理迁移版本，每个 migration 有对应回滚脚本

**前端构建失败无回滚方案**
- 低优先级，不阻断交付，可在阶段 5 补充
